The CLAUDE.md Cheat Sheet

Your highest-leverage file. One bad line cascades into hundreds of bad lines of code. Treat it like production code. Ruthlessly edit it over time.

To learn more, go to masterclaudecode.com
1 bad CLAUDE.md line → many bad research lines → many bad plan lines → 100s of bad code lines

✓ Golden Rules

  • Start small. Begin with just a project description + key commands. Add lines only when the model makes a specific mistake.
  • Remove more than you add. With every model upgrade, look at what you can delete. Newer models have better practices baked in.
  • Commit every change. Git-track your CLAUDE.md so you can pinpoint which line caused a regression.
  • Don't do the model's thinking for it. Over-constraining a capable model leads to worse results (the Vercel DZero lesson).
  • Audit regularly. Weekly or after each model release. Look for conflicts, stale rules, and lines that belong in sub-files.

✗ Anti-Patterns to Avoid

  • Copy-pasting random prompts from Twitter/GitHub into your file. If you didn't write it for your project, it probably hurts.
  • Using it as a changelog. It's not a history log. It's a live specification.
  • 500+ line files. ~10% of public CLAUDE.md files exceed 500 lines. Models degrade past ~150-250 instructions.
  • Teaching obvious best practices. "Use encryption for passwords" or example code snippets waste your instruction budget.
  • Never removing anything. Keeping rules for old model quirks constrains newer, more capable models.
  • Auto-generating with /init and leaving it untouched. Auto-generated files are too verbose by default.

☰ Ideal Root CLAUDE.md Structure

Position matters. LLMs weigh the beginning and end of instructions more heavily than the middle.

# Project Description ← FIRST: what this project is & how it all fits together
Brief 2-3 sentence overview of the app/system

# Key Commands ← things NOT inferable from the codebase
npm run dev, npm test, deployment commands, etc.

# Architecture Notes ← only if genuinely non-obvious
Major patterns, key abstractions, data flow

# Caveats & Hard-Won Lessons ← things the model keeps getting wrong
Specific behavioral corrections added one at a time

📂 Split Into Nested Files

  • Root = lightweight. Only globally relevant info. Loaded on every single request.
  • Nested = context-heavy. Domain-specific rules in subfolders. Lazy-loaded only when files in that folder are read.
  • How it works: When Claude reads a file, any CLAUDE.md in the same directory (and parents) is appended to the tool result.
  • Example: Move Supabase migration rules from root into supabase/CLAUDE.md. Only injected when working on migrations.
  • Benefit: Right context at the right time, without bloating every conversation.

🛡 Hooks vs CLAUDE.md

  • CLAUDE.md = guidance. The model may ignore it 1 in 50 times, especially in long or confused sessions.
  • Hooks = enforcement. pre_tool_use hooks block commands every single time, guaranteed.
  • Rule of thumb: If something must never happen (like db push), use a hook. If it's a preference, use CLAUDE.md.
  • Bonus: Moving critical constraints to hooks frees up instruction budget in your CLAUDE.md file.
  • Examples: Block dangerous CLI commands, prevent edits to protected folders, enforce linting before commit.

↻ The Model Upgrade Cycle

1 New model drops It likely handles edge cases you wrote rules for
2 Test without rules Try a clean session and see what breaks
3 Remove what's fixed Delete rules the model no longer needs
4 Add only new gaps Only add rules for new mistakes you observe

✓ What Belongs in CLAUDE.md

  • Project description & high-level architecture
  • Non-obvious commands & workflows
  • Hard-won lessons from actual model mistakes
  • Project-specific conventions the model can't infer
  • Pointers to key files or unusual patterns

✗ What Doesn't Belong

  • Generic coding best practices
  • Example code snippets for common patterns
  • Things the model already knows how to do
  • Historical notes / changelog entries
  • Critical safety rules (use hooks instead)
