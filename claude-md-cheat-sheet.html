The CLAUDE.md Cheat Sheet

Everything you need to know about writing, structuring, and maintaining CLAUDE.md files for maximum agent performance.

From masterclaudecode.com

01 The Big Idea

The bottleneck isn't intelligence. It's context. The gap between what AI can do and what you're getting comes down to one thing: did the model have the right information before it made any decisions? CLAUDE.md is your primary tool for closing that gap.

Signal-to-Noise

Every token in context affects every other token. Noise drowns signal. Your job is to curate the smallest, highest-signal context for each task.

You Only Get The Behavior You Load

If the model doesn't see specific context, it fills the gap with generic training priors. Custom patterns, historical decisions, tribal knowledge — none of it exists unless you load it.

Search Isn't Enough

Agentic search fills context with dead ends before finding signal. Code doesn't capture intent, history, or tribal knowledge. Context engineering hands the agent a map.

02 Why Naive Approaches Fail

Instruction Following Has Limits Research

IFScale benchmark (July 2025): even frontier models hit only 68% accuracy at 500 instructions. Best models perform near-perfect until ~150 instructions, then fall off a cliff.

  • Instructions compete with each other
  • Important rules get lost in noise
  • The model drops constraints silently
  • Behavior becomes inconsistent across sessions

Source: arxiv.org/abs/2507.11538

Cognitive Inertia Research

Models resist patterns that diverge from training. They've seen millions of conventional examples and will "correct" your unconventional patterns back to standard.

  • Simple instructions won't overcome it
  • You need concrete code examples, not prose
  • The Inverse IFEval benchmark shows models perform significantly worse on counter-intuitive instructions
  • Work with existing patterns when possible

Source: arxiv.org/abs/2509.04292

The System Reminder Band-Aid Claude Code wraps every CLAUDE.md file with: "This context may or may not be relevant to your tasks." This was added because most CLAUDE.md files are stuffed with irrelevant instructions. If your file is good, the wrapper is a no-op. If it's bad, the model may ignore your instructions entirely. Write a CLAUDE.md that doesn't need filtering.

03 The Context Layer

The solution is progressive disclosure: load only what you need, when you need it. The context layer has three components:

Layer Nodes

CLAUDE.md / AGENTS.md files placed at semantic boundaries. Auto-load when agent enters that directory. Root is always loaded; descendants load on demand.

Skills

Progressively disclosed capabilities. Agent sees metadata (10-100 tokens), loads full skill only when task matches. Good for framework patterns, complex workflows, domain knowledge.

Hooks

Contextual automation at specific points. Pre-tool hooks for validation (block destructive commands). Post-tool hooks for linting. More reliable than CLAUDE.md rules for enforcement.

project/
├── CLAUDE.md ← Always loaded (root — keep minimal!)
├── services/
│   ├── CLAUDE.md ← Loaded when working in services/
│   ├── payment/
│   │   └── CLAUDE.md ← Loaded when working in payment/
│   └── billing/
│       └── CLAUDE.md ← Loaded when working in billing/
└── frontend/
    └── CLAUDE.md ← Loaded when working in frontend/
The T-Shaped View When an agent works on a file, it gets horizontal context (ancestors auto-load up the tree) and vertical depth (specific detail for the current area). A task in /api/auth/ loads root + api + auth nodes. It never loads frontend or scripts. Exactly what it needs, nothing more.

04 The Six Core Areas

You don't need all six. But if your CLAUDE.md doesn't touch any of them, it's probably not adding value.

1. Commands

Put these early in the file. Include full flags and options, not just tool names.

# Type check a single file
npm run tsc --noEmit path/to/file.tsx

# Run single test file
npm run vitest run path/to/file.test.tsx

2. Testing

How to run tests, what patterns to use, where test files live.

Tests live alongside source as *.test.ts.
Use vi.mock() for mocking.
Run: pnpm test:unit <path>
E2E requires docker compose up first.

3. Project Structure

What lives where. Keep it abstract enough to survive refactors. Remove obvious things.

apps/web — Next.js frontend
apps/api — Express backend
packages/shared — Shared types
packages/ui — Component library

4. Code Style

Via examples, not prose. Point to files the agent should copy.

- Prefer functional components like Projects.tsx
- Forms: copy app/components/DashForm.tsx
- HTTP: use app/api/client.ts — never
  fetch directly in components

5. Git Workflow

Especially if it differs from standard practice.

Branch: feature/JIRA-123-description
Always rebase on main before merging.
Squash commits in PR.
Never push directly to main.

6. Boundaries

What should never be touched. Consider also enforcing via hooks.

- Never commit secrets or .env files
- Don't modify vendor/ or node_modules/
- Production configs in deploy/ off-limits
- Don't touch legacy/ unless asked

05 What to Include vs. Exclude

The Litmus Test For each line, ask: "If I removed this, would Claude do something wrong on most tasks?" If no, it probably doesn't belong in root.

Example 2: The Bug Fix Entry (80 lines → 15 lines)

BEFORE (~80 lines)

Full crash signature, stack traces, 3 code examples, tutorial-length explanation of actor isolation, alternative approaches, historical background…

Problem: Core rule fits in 2 sentences. The rest is tutorial content. Crash signatures aren't actionable for prevention.

AFTER (~15 lines)

### SwiftUI + @MainActor Singleton Crash

Never use @StateObject with a @MainActor
singleton. Use @ObservedObject instead.

// WRONG — crashes with EXC_BAD_ACCESS
@StateObject private var monitor = MyMonitor.shared

// CORRECT — view observes, doesn't own
@ObservedObject private var monitor = MyMonitor.shared

Why: @StateObject is for objects the view
owns. Singletons aren't owned by any view.

08 The Five Golden Rules

1
Specific over Generic. Project details, not universal truths. "Never bypass requireAuth" beats "use proper auth."
2
Commands over Descriptions. Things Claude can actually run. Full flags, not just tool names.
3
Why over What. Rationale that informs decisions. "Audit logs depend on req.user" — the agent can apply this reasoning elsewhere.
4
Pointers over Explanations. See src/auth/AuthService.ts stays accurate as code changes. Inline prose rots.
5
Ruthlessly Edited. 15 lines over 80. When in doubt, cut it. A 230-line file can usually compress to 30–40 lines.
"The best layer nodes will be specific over generic, give commands over descriptions, give a why over what, use pointers over explanations, and be ruthlessly edited to be as nimble and lean as possible."

10 The Hierarchy System

Three Levels of CLAUDE.md

  • ~/.claude/CLAUDE.md — Enterprise/personal global rules (coding standards, compliance). Loads everywhere.
  • ./CLAUDE.md — Project rules (architecture, conventions). Share via Git.
  • .claude/CLAUDE.local.md — Personal preferences (gitignored). "I'm a beginner with TypeScript."

They load in order, higher levels first. Within each file, rules read top to bottom in priority — put most important rules at the top.

Where to Place Nodes

Place at semantic boundaries, not every directory:

  • Where responsibility shifts
  • Where contracts between components matter
  • Where specialized context is required
  • Where the agent repeatedly makes mistakes

Don't put one in every single directory. That's over-engineering.

@imports System

Keep CLAUDE.md lean. Pull in detail from other files:

See @README.md for project overview
See @docs/api-patterns.md for API conventions
See @package.json for available npm scripts

Imports can be recursive. Use sparingly to avoid creating a maze.

The LCA Principle

When a fact applies to multiple areas, place it at the Least Common Ancestor — the shallowest node covering all relevant paths.

  • Not in each leaf: duplication drifts out of sync
  • Not deeper than necessary: loads when irrelevant
  • At LCA: loads exactly when needed

Ask Claude to update your CLAUDE.md — never edit manually. When Claude makes a mistake, fix it manually, then tell Claude "update the rules so we never do this again." Every mistake becomes a permanent fix. Your CLAUDE.md evolves like lint rules shaped by real sessions.

12 Maintenance

The "Would It Mislead?" Test

For each change, ask: "If an agent read the current node and worked on this code, would it make a mistake?"

Update when: new patterns, changed invariants, changed contracts, new gotchas, node would mislead.

Skip when: implementation details change but contracts don't, bug fixes, refactors that don't change responsibilities.

Leaf-First Updates

When code changes, update nodes leaf-first, working up the tree. Parent nodes summarize children — if you update a parent before children, you're summarizing stale content.

Change in payment/validators/amount.ts

Update order:
1. payment/validators/CLAUDE.md
2. payment/CLAUDE.md
3. services/CLAUDE.md
4. Root (only if system-wide impact)
Code changes → Nodes updated → Agents use context → Agents surface gaps → Learnings fed back → Better context → (repeat)

Manual (Start Here)

5–10 min per significant PR. Review which nodes might be affected. Read diff alongside node. Update if it would mislead.

Semi-Automated

Script detects changed files, maps to covering nodes, generates PR with suggested updates. Human reviews.

Fully Automated

Agent monitors merges, proposes updates, runs through CI. Human approves. The endgame.

New Model = Remove Stuff Every time a new model releases, your first instinct should be to remove instructions, not add them. Smarter models need less hand-holding. Instructions that helped older models become noise for newer ones.

13 Session & Context Window Tips

Session-End Prompt

At the end of significant sessions, ask:

"Based on this session, should any CLAUDE.md files be updated?"

Good candidates: gotchas discovered, commands run multiple times, patterns Claude kept getting wrong.

Bad candidates: task-specific context, explanations of what you just built, temporary workarounds.

Context Window Discipline

  • Stay under 50% of available context for best results
  • Don't context switch between unrelated tasks
  • Group related tasks in the same session
  • Compact at most once, then start fresh
  • Plan in one session, execute in another
  • Write bug fix attempts to a file before new chat
  • Use /compact with focus instructions

14 Advanced Techniques

One Pattern Per Thing

If your CLAUDE.md says "use Result types" but half the codebase uses try/catch, the agent gets contradictory signals. The code is a stronger signal than the doc.

Fix: Use subagents to audit for conflicting patterns. Pick a winner. Migrate. Then delete the CLAUDE.md rule — a clean codebase with one pattern is self-documenting.

"CLAUDE.md rules are compensating for codebase inconsistency. The goal is to not need them."

Delete Your README

README = third source of truth (alongside CLAUDE.md and code). It has no feedback loop — nobody notices when it's wrong.

Fix: Delete it. When you need docs, spawn an Explore subagent to generate fresh context from the actual codebase. Just-in-time context beats stale documentation.

Overcoming Cognitive Inertia

When your codebase uses unconventional patterns:

  • Create a CLAUDE.md in the relevant folder with 3 patterns and anti-patterns with code examples
  • Or turn it into a skill (loads on demand vs. always loaded)
  • Concrete code examples beat prose every time
  • Isolate unconventional work to separate contexts

Triggering Skills Reliably

Vercel found relying on the agent to decide when to trigger a skill was unreliable. Their fix:

  • Compress documentation into a tiny index in the AGENTS.md file (~2,200 tokens)
  • Add to the relevant layer node: "Prefer retrieval-led reasoning over pre-training-led reasoning"
  • Put knowledge-gap skills in the relevant subfolder node, not root

15 The Audit Prompt

Paste this into Claude Code to restructure your context layer:

Audit my context layer and help me restructure it for progressive disclosure.

1. Find all CLAUDE.md files. List them with locations and line counts.
2. Find contradictions — instructions that conflict within or across files.
3. Identify essentials for root (one-sentence description, commands, universal invariants).
   Flag everything else for relocation.
4. Identify semantic boundaries where new CLAUDE.md files should be created.
   Also identify content for skills and hooks.
5. Apply the LCA principle — shared facts at shallowest covering node.
6. Flag for deletion: redundant, vague, obvious, or linter-enforced rules.
   Ask me to confirm each deletion.

The Quality Checklist

For each line, ask:

  • Is this project-specific? (If universal, delete it)
  • Would an agent go wrong without this? (If not, it's optional)
  • Does the code already show this? (If yes, don't repeat it)
  • Will this rot? (If yes, phrase abstractly or link to source)
  • Is this the right level? (Root = everywhere, Leaf = here only)

Signs the layer is healthy:

  • Agents find relevant files on first try
  • Fewer "I couldn't find X" moments
  • New team members productive faster
  • Agents respect existing patterns
  • Rarely correct the same misunderstanding twice
  • Each task leaves the layer slightly better
